# 接入中国大陆券商的小资金非高频量化系统方案

## 1. 总目标

构建一个能真实接入中国大陆券商的小资金非高频量化系统。系统应优先满足：

- **可接入**：执行器通过明确的券商适配器边界对接 QMT、Ptrade、券商开放 API 或其他经授权终端。
- **可控风险**：所有策略信号必须先经过交易规则、账户约束和风控阈值过滤。
- **可审计**：保留从行情快照、信号、约束裁剪、订单提交、成交回报到人工干预的链路。
- **可渐进实盘**：按研究回测、纸面交易、仿真柜台、小额实盘、规模化实盘逐步推进。

## 2. 券商接入现实路径

中国大陆个人或小资金量化实盘通常无法直接连交易所，常见路径如下：

| 路径 | 适合阶段 | 优点 | 主要风险 |
| --- | --- | --- | --- |
| QMT / XtQuant | 仿真到实盘 | 生态成熟、覆盖常见交易动作 | 版本差异、终端登录稳定性、券商授权差异 |
| Ptrade | 仿真到实盘 | 策略托管和终端集成较完整 | 平台规则约束、调试链路较黑盒 |
| 券商官方 API | 实盘 | 合规边界清晰、账户权限明确 | 开通门槛和功能范围依券商而定 |
| 手工确认半自动 | 早期实盘 | 风险最低、便于验证信号 | 执行效率低，无法完全闭环 |

系统设计上应把“券商适配器”视为可替换边界，不让策略代码依赖某个具体终端 SDK。

## 3. 最小可行架构

```text
行情源 -> 特征计算 -> 策略信号 -> 约束引擎 -> 订单执行器 -> 券商适配器
                         |              |              |
                         v              v              v
                       风控检查       审计日志       成交回报
```

### 3.1 策略层

策略层只负责输出目标权重或交易意图，例如：

- 单票目标权重不超过 20%。
- 不主动交易停牌、ST、退市整理或流动性不足标的。
- 日频策略在收盘后生成次日计划，盘中只做异常处理。

### 3.2 约束层

约束层是实盘前的硬闸门，至少包括：

- A 股股票买入 100 股整数倍，卖出可按持仓零股规则处理。
- T+1 股票当日买入不可当日卖出；ETF、可转债等规则单独配置。
- 涨停不追买、跌停不强制卖出，除非人工或特殊风控策略确认。
- 单笔订单金额、单日成交额、单票仓位、总仓位上限。
- 现金预留，覆盖手续费、滑点和最低佣金。

### 3.3 执行层

执行器负责把约束后的订单转换成券商接口调用：

- 提交前记录 `client_order_id`。
- 提交后轮询或订阅订单状态。
- 对部分成交、废单、撤单失败做状态机处理。
- 遇到连接异常时进入安全模式，禁止重复盲目下单。

### 3.4 风控层

小资金系统常见风险不是模型复杂度不足，而是执行和约束缺失。建议默认启用：

- 最大单日亏损：如账户权益的 2%。
- 最大回撤停机：如从近期高点回撤 8% 后只允许减仓。
- 单票最大仓位：如 20%。
- 单日最大换手：如账户权益的 100%。
- 异常价格保护：订单价格偏离最新价超过阈值时拒单。

## 4. 从零到实盘路线图

1. **第 0 阶段：规则建模**
   - 明确交易品种、交易时间、费用、最小单位和 T+1/T+0 规则。
   - 建立本文档中的约束引擎，并用单元测试固定规则。
2. **第 1 阶段：离线研究**
   - 使用复权行情、财务/指数成分等数据做策略研究。
   - 将策略输出限制为目标仓位，避免研究代码直接包含下单逻辑。
3. **第 2 阶段：回测与纸面交易**
   - 引入手续费、最低佣金、滑点、涨跌停不可成交、停牌等假设。
   - 对每日计划生成审计报告。
4. **第 3 阶段：仿真柜台**
   - 接入券商仿真环境或终端模拟账户。
   - 验证订单状态机、断线恢复、重复下单保护。
5. **第 4 阶段：小额实盘灰度**
   - 从极小仓位开始，只允许白名单标的。
   - 每日人工复核成交和持仓差异。
6. **第 5 阶段：稳定实盘**
   - 自动化盘前检查、盘中监控、盘后归因。
   - 保留人工一键停机和只减仓模式。

## 5. 当前代码原型

`src/kelaode/core.py` 提供了最小闭环：

- `OrderIntent` 表示策略意图。
- `TradingConstraints` 表示账户和市场硬约束。
- `ConstraintEngine` 将意图裁剪为合规订单。
- `PaperBrokerAdapter` 作为券商适配器占位实现。
- `ExecutionEngine` 统一执行、记录订单并处理拒单原因。

后续接入真实券商时，应新增独立 adapter，例如 `QmtBrokerAdapter`，并保持核心策略和约束层不变。
